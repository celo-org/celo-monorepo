## Overview

The Celo Core Contracts release process ensures that smart contract upgrades are deployed safely through proper versioning, testing, and governance procedures. This guide covers the Foundry-based tooling that replaces the legacy Truffle-based release scripts.

Core contract releases are rolled out via Celo's on-chain governance system, with approximately four major releases per year.

### Script Mapping (Truffle → Foundry)

| Truffle Command | Foundry Equivalent |
|-----------------|-------------------|
| `release:check-versions` | `release:check-versions:foundry` |
| `release:verify-deployed` | `release:verify-deployed:foundry` |
| `release:make` | `release:make:foundry` |
| `release:verify-release` | *(use verify-deployed with --proposal)* |

## Versioning

Each Celo core smart contract is versioned independently using **semantic versioning** with a custom extension:

| Version Type | Description |
|-------------|-------------|
| **STORAGE** | Incompatible storage layout changes |
| **MAJOR** | Incompatible ABI changes |
| **MINOR** | Added functionality (backwards compatible) |
| **PATCH** | Backwards-compatible bug fixes |

All deployed core contracts implement `getVersionNumber()` which returns `(storage, major, minor, patch)` encoded in Solidity source. Contracts deployed before versioning was added default to version `1.1.0.0`.

**Important**: If mixins or libraries change, all contracts using them are considered changed and must be redeployed in the next release.

## GitHub Branching & Tagging

### Branch Naming Convention

Release development happens on branches named:
bash
release/core-contracts/${N}

Where `N` is the release version number (e.g., `release/core-contracts/14`).

## Release Artifacts

The release process generates and uses several important artifacts. Understanding these is crucial for a successful release.

### 1. `libraries.json` - Library Address Mapping

**What it is:** A JSON file containing the deployed addresses of all linked libraries on a network.

**How it's generated:** Automatically created by `release:verify-deployed:foundry` after successfully verifying on-chain bytecode.

**When to generate:**
bash
# Generate libraries.json for Celo Sepolia
yarn release:verify-deployed:foundry -b core-contracts.v${PREVIOUS} -n celo-sepolia -f

# Generate libraries.json for Mainnet
yarn release:verify-deployed:foundry -b core-contracts.v${PREVIOUS} -n celo -f


**Output location:** `./libraries.json` (in the `packages/protocol` directory)

**Example content:**

{
  "Proposals": "0x38afc0dc55415ae27b81c24b5a5fbfe433ebfba8",
  "IntegerSortedLinkedList": "0x411b40a81a07fcd3542ce5b3d7e215178c4ca2ef",
  "AddressLinkedList": "0xd26d896d258e258eba71ff0873a878ec36538f8d",
  "Signatures": "0x69baecd458e7c08b13a18e11887dbb078fb3cbb4",
  "AddressSortedLinkedList": "0x4819ad0a0eb1304b1d7bc3afd7818017b52a87ab"
}


**Why it's needed:** The `release:make:foundry` script uses these addresses to link libraries when deploying and verifying updated contracts. Keep the generated file for the network you are releasing to, and regenerate it if you switch networks or branches.

### 2. Release Transaction JSON

**What it is:** A JSON file containing the governance proposal transactions generated by `release:make:foundry`.

**How it's generated:**
bash
yarn release:make:foundry -b core-contracts.v${PREVIOUS} -n <network>


**Why it's needed:** This file is used to review the exact governance transactions that will be proposed and later to verify that deployed contracts match the expected release output.

### 3. Extra Transactions JSON

**What it is:** An optional JSON file containing additional governance transactions that should be appended to the release proposal and included during verification.

**When to use it:** Use `--extraTxs` when a release requires transactions that are not generated automatically by `release:make:foundry`, such as one-off configuration changes, registry updates, parameter changes, or operational transactions that must be included in the same governance proposal.

**How to use it with release generation:**
bash
yarn release:make:foundry -b core-contracts.v${PREVIOUS} -n <network> --extraTxs ./extra-txs.json


**How to use it with release verification:**
bash
yarn release:verify-deployed:foundry -b core-contracts.v${RELEASE} -n <network> --proposal ./releaseData/transactions.json --extraTxs ./extra-txs.json


**Important:** If `--extraTxs` is used when generating a release, pass the same `--extraTxs` file to `release:verify-deployed:foundry` when verifying the proposal. This ensures the verifier checks the full set of transactions that operators intend to submit through governance.

## Release Commands

### Check Contract Versions

Use this command to confirm that changed contracts have had their version numbers updated correctly:
bash
yarn release:check-versions:foundry -b core-contracts.v${PREVIOUS}


### Verify Deployed Contracts

Use this command to verify that deployed contracts on a network match the expected bytecode for a branch or tag:
bash
yarn release:verify-deployed:foundry -b core-contracts.v${VERSION} -n <network>


Common options:

| Option | Description |
|--------|-------------|
| `-b, --branch` | Git branch or tag to compare against |
| `-n, --network` | Target network, such as `celo-sepolia` or `celo` |
| `-f, --fetch` | Fetch deployed library addresses and write `libraries.json` |
| `--proposal <path>` | Verify a generated governance proposal transaction file |
| `--extraTxs <path>` | Include additional transactions when verifying a proposal |

Examples:
bash
# Verify the previous release and generate libraries.json
yarn release:verify-deployed:foundry -b core-contracts.v${PREVIOUS} -n celo-sepolia -f

# Verify a release proposal, including extra transactions
yarn release:verify-deployed:foundry -b core-contracts.v${RELEASE} -n celo-sepolia --proposal ./releaseData/transactions.json --extraTxs ./extra-txs.json


### Make Release

Use this command to generate the deployment and governance proposal transactions for a release:
bash
yarn release:make:foundry -b core-contracts.v${PREVIOUS} -n <network>


Common options:

| Option | Description |
|--------|-------------|
| `-b, --branch` | Previous release branch or tag used as the baseline |
| `-n, --network` | Target network |
| `--extraTxs <path>` | Append additional governance transactions to the generated release proposal |

Examples:
bash
# Generate a standard release
yarn release:make:foundry -b core-contracts.v${PREVIOUS} -n celo-sepolia

# Generate a release with additional governance transactions
yarn release:make:foundry -b core-contracts.v${PREVIOUS} -n celo-sepolia --extraTxs ./extra-txs.json


When using `--extraTxs`, review the generated proposal output carefully to confirm that the extra transactions appear in the expected order and target the intended contracts.

## Step-by-Step Release Process

1. **Create or update the release branch**
bash
   git checkout -b release/core-contracts/${N}


2. **Update contract versions**
   - Update `getVersionNumber()` for every changed contract.
   - Ensure dependent contracts are versioned if their inherited contracts, mixins, or linked libraries changed.

3. **Run tests**
bash
   yarn test
   forge test


4. **Check versions**
bash
   yarn release:check-versions:foundry -b core-contracts.v${PREVIOUS}


5. **Verify the currently deployed release and generate `libraries.json`**
bash
   yarn release:verify-deployed:foundry -b core-contracts.v${PREVIOUS} -n <network> -f


6. **Prepare any additional governance transactions, if needed**
   - Create an `extra-txs.json` file containing the additional transactions.
   - Review each transaction target, value, calldata, and ordering.
   - These transactions must be supplied to both release generation and release verification using `--extraTxs`.

7. **Generate the release**
bash
   yarn release:make:foundry -b core-contracts.v${PREVIOUS} -n <network>


   If the release includes additional transactions:
bash
   yarn release:make:foundry -b core-contracts.v${PREVIOUS} -n <network> --extraTxs ./extra-txs.json


8. **Review release artifacts**
   - Confirm the set of contracts to deploy.
   - Confirm linked library addresses.
   - Confirm generated governance proposal transactions.
   - If `--extraTxs` was used, confirm the extra transactions are included and ordered correctly.

9. **Verify the generated release proposal**
bash
   yarn release:verify-deployed:foundry -b core-contracts.v${RELEASE} -n <network> --proposal ./releaseData/transactions.json


   If the release includes additional transactions, pass the same file used during release generation:
bash
   yarn release:verify-deployed:foundry -b core-contracts.v${RELEASE} -n <network> --proposal ./releaseData/transactions.json --extraTxs ./extra-txs.json


10. **Submit the governance proposal**
    - Submit the verified transaction list through the governance process.
    - Include any additional transactions that were supplied with `--extraTxs`.

11. **Verify after execution**
bash
    yarn release:verify-deployed:foundry -b core-contracts.v${RELEASE} -n <network>


## Release Timeline

A typical release should include:

- Release branch creation and version updates.
- Test and version verification.
- Generation of network-specific release artifacts.
- Review of generated governance transactions.
- Governance proposal submission.
- Voting, approval, queueing, and execution.
- Post-execution verification.

## Testing & Verification

Before submitting a release proposal:

- Run unit and integration tests.
- Run `release:check-versions:foundry`.
- Run `release:verify-deployed:foundry` against the previous deployed release.
- Run `release:make:foundry` to generate release transactions.
- Run `release:verify-deployed:foundry --proposal` against the generated proposal.
- If extra governance transactions are part of the release, include `--extraTxs <path>` in both `release:make:foundry` and `release:verify-deployed:foundry`.

## Governance Proposal

The generated release transactions are intended to be submitted through Celo governance. Operators should review:

- Target addresses.
- Transaction values.
- Calldata.
- Ordering of transactions.
- Contract implementation addresses.
- Registry updates.
- Proxy upgrades.
- Any additional transactions supplied through `--extraTxs`.

If the release includes manually authored transactions, keep the `extra-txs.json` file with the release artifacts so reviewers and verifiers can reproduce the complete proposal.

## Communication & Community

Communicate planned releases to validators, governance participants, and community stakeholders. Include:

- Release scope.
- Contracts affected.
- Governance proposal links.
- Any operational requirements.
- Any extra transactions included in the proposal.

## Emergency Patches

For emergency patches:

- Keep the scope minimal.
- Update versions only for affected contracts and dependencies.
- Run the same verification steps as a regular release whenever possible.
- If emergency governance actions require additional transactions, include them with `--extraTxs` in both release generation and verification.

## Troubleshooting

### `libraries.json` is missing or stale

Regenerate it for the correct network and baseline release:
bash
yarn release:verify-deployed:foundry -b core-contracts.v${PREVIOUS} -n <network> -f


### Proposal verification fails when extra transactions are expected

Confirm that the same extra transactions file was passed to both commands:
bash
yarn release:make:foundry -b core-contracts.v${PREVIOUS} -n <network> --extraTxs ./extra-txs.json

yarn release:verify-deployed:foundry -b core-contracts.v${RELEASE} -n <network> --proposal ./releaseData/transactions.json --extraTxs ./extra-txs.json


Also confirm that the file has not changed between generation and verification.

### Bytecode verification fails

- Confirm you are using the correct branch or tag.
- Confirm the correct network is selected.
- Regenerate `libraries.json` for the network.
- Confirm linked libraries match the deployed addresses.

### Version check fails

- Ensure every changed contract has an updated version number.
- Ensure contracts using changed mixins or libraries have also been updated.

## Deployer Keys

Ensure the deployer account and any governance proposer account are configured for the target network. Never commit private keys or sensitive environment files.

## Starting a Local Fork

A local fork can be useful for testing release generation and verification:
bash
anvil --fork-url $CELO_RPC_URL


Then run the relevant release commands against the fork configuration used by your environment.

## Contract Verification (Block Explorers)

After deployment or governance execution, verify updated contract implementations on the appropriate block explorer. Ensure compiler settings, optimizer settings, linked libraries, and constructor arguments match the generated artifacts.

## Environment Variables

Configure the environment variables required by the release scripts and Foundry before running release commands. Common variables include:

| Variable | Description |
|----------|-------------|
| `CELO_RPC_URL` | RPC endpoint for Celo Mainnet |
| `CELO_SEPOLIA_RPC_URL` | RPC endpoint for Celo Sepolia |
| `PRIVATE_KEY` | Deployer or proposer private key, where required |

Do not commit secrets or local environment files.
